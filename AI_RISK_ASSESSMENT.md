# Risk Assessment: AI-Generated Code Base

Since this application was generated by AI, there are specific classes of risks that differ from human-written legacy code. AI tends to be "correct locally, but fragile globally".

## 1. The "Happy Path" Bias
**Risk:** AI assumes external services (Stripe, Database, Redis) are always fast and available.
**Finding:**
*   **Race Conditions in Quotas:** `QuotaService.checkQuota` fetches current usage (`usageService.getFeatureUsage`) and *then* verifies.
    *   *Scenario:* A user sends 10 parallel requests. All 10 read `currentUsage: 49`. Limit is `50`. All 10 pass. Final usage: `59/50`.
    *   *Impact:* Users can exceed limits significantly if they exploit concurrency.
    *   *Fix:* Use Redis `INCR` or Database atomic transactions logic for quota enforcement.

## 2. "Magic String" Logic
**Risk:** AI often relies on naming conventions rather than explicit configuration, making refactoring dangerous.
**Finding:**
*   **Inferred Feature Keys:** `quota.service.ts` contains logic like:
    ```typescript
    if (serviceSlug.includes('ai')) inferredFeatureKey = 'ai_generation';
    ```
    *   *Danger:* If you rename a service to `smart-writer` (no "ai"), this logic fails silently, and the service becomes **free and unlimited**.
    *   *Fix:* Explicitly define `requiredFeatureKey` in the Database for *every* service. Remove the inference fallback.

## 3. Dependency Hallucination & Bloat
**Risk:** AI tends to grab "popular" libraries even if a native function would do, or mixes incompatible versions.
**Finding:**
*   **Mixed Auth Practices:** You have `passport` (standard) but also `bcryptjs` and manual session handling disjoint from it in some places. Ensure one source of truth for user identification.
*   **Version Pinning:** Check `package.json`. AI often picks whatever version was in its training data. You are using `@prisma/client` ^6.0.0 (very new) which is good, but verify that `scripts/` use compatible node versions.

## 4. Operational Blind Spots
**Risk:** AI implements *features* but forgets *maintenance*.
**Finding:**
*   **Log Rotation:** `pino` is used, but is there a log rotation strategy? If `app.log` grows to 50GB, the server crashes.
*   **Database Cleanup:** `usageService` logs every API request to the database. Over a year, the `UsageLog` table will contain millions of rows, slowing down the `checkQuota` query (which effectively does a `COUNT *`).
    *   *Impact:* System gets slower over time until it times out.
    *   *Fix:* Implement data retention policies (e.g., "Delete logs older than 90 days") or use optimized analytics stores (TimescaleDB, ClickHouse) instead of the primary Postgres DB for logs.

## 5. Security Edge Cases
**Finding:**
*   **Token Generation:** `AuthService` uses `Math.random()` for some codes.
    ```typescript
    const code = Math.floor(100000 + Math.random() * 900000).toString();
    ```
    *   `Math.random` is **not cryptographically secure**. It is predictable.
    *   *Impact:* An attacker can predict 2FA codes if they know the server time and state.
    *   *Fix:* Use `crypto.randomInt()` or `crypto.randomBytes()`.

## Summary
The code is cleaner than average "spaghetti code", but it is **fragile**. It relies on everything working exactly as named. A human developer might rename a variable and break the billing logic because of the "magic string" dependency.

**Immediate Actions:**
1.  **Fix the Randomness:** Replace `Math.random` with `crypto` for 2FA.
2.  **Harden Quotas:** Accept that race conditions exist, or move to atomic increments.
3.  **Monitor Database Size:** Watch the `UsageLog` table size carefully.
